## Architectural Verdict

Do not fully remove the old module in the first implementation slice. Supersede-in-place is better for the owner, even though the owner asked for removal.

The reason is data safety and deployment continuity. Migration 024 may already be applied to live Postgres and creates eight tables plus RLS policies and dark flags (`migrations/024_enterprise_programme_foundation.sql:110`, `migrations/024_enterprise_programme_foundation.sql:125`, `migrations/024_enterprise_programme_foundation.sql:140`, `migrations/024_enterprise_programme_foundation.sql:164`, `migrations/024_enterprise_programme_foundation.sql:179`, `migrations/024_enterprise_programme_foundation.sql:208`, `migrations/024_enterprise_programme_foundation.sql:228`, `migrations/024_enterprise_programme_foundation.sql:250`, `migrations/024_enterprise_programme_foundation.sql:333`, `migrations/024_enterprise_programme_foundation.sql:350`, `migrations/024_enterprise_programme_foundation.sql:366`, `migrations/024_enterprise_programme_foundation.sql:372`, `migrations/024_enterprise_programme_foundation.sql:378`, `migrations/024_enterprise_programme_foundation.sql:384`, `migrations/024_enterprise_programme_foundation.sql:390`, `migrations/024_enterprise_programme_foundation.sql:396`). It is dark by default (`migrations/024_enterprise_programme_foundation.sql:415`), so the operational risk of leaving it disabled is low. Deleting the code and dropping the tables first would destroy any pilot programme rows, organisation bootstrap records, project links, and audit rows without adding lifecycle value.

The right owner-protective decision is: keep the existing flag dark, replace the implementation behind the same `/enterprise` surface and flag, migrate/extend the old tables where useful, add new lifecycle/gate/workflow tables, and only drop obsolete columns/tables after a backup, export, validation report, and one release of compatibility reads. Full physical removal can be a later cleanup milestone once live data is proven empty or migrated.

## Final Teardown Recommendation

Release 1 should supersede in place and migrate forward. Full removal is worse for the owner today because it spends risk budget on deleting dark scaffolding instead of implementing the missing operational spine. The old module is not fit for purpose, but it contains useful compatibility seams: `/enterprise`, the feature flag, membership bootstrap, programme/project link semantics, and audit precedent. Keep those seams, replace the internals, and reserve destructive deletion for a post-migration cleanup release.
