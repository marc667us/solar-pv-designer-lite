SolarPro Enterprise Solar Programme Management Module

You are operating as two coordinated senior software-engineering systems:

OpenAI Codex — Architecture, repository analysis and implementation planning authority
Claude Code — Implementation, integration, testing, remediation and deployment authority

Your joint assignment is to extend the existing SolarPro application by adding a production-grade Enterprise Solar Programme Management Module.

This is not a request to create a new application, a separate prototype, a disconnected micro-product or a replacement for the existing SolarPro platform.

The new capability must be implemented as a native, securely integrated module that reuses existing SolarPro services, data models, workflows, engineering engines, AI agents, dashboards and enterprise controls wherever technically appropriate.

The module must support national, regional, community, institutional, commercial, agricultural, residential and utility-scale solar programmes involving tens, hundreds or thousands of individual SolarPro projects.

1. PRIMARY OPERATING RULE

Codex must first inspect the existing repository and prepare a detailed implementation plan.

Claude Code must then accept that implementation plan as the controlling delivery specification, validate it against the actual repository, improve it where technically necessary, and execute the work through development, integration, testing and deployment.

Neither system may assume the application architecture, framework, database, authentication system, folder structure, deployment provider or existing feature names without inspecting the repository.

Do not rebuild working capabilities.

Do not duplicate existing modules.

5. NON-NEGOTIABLE DESIGN PRINCIPLES

The implementation must observe these principles:

5.1 Native Module

The new feature must appear inside the existing SolarPro navigation, application shell, permissions system and user experience.

5.2 Reuse Before Creation

Codex must identify existing reusable capabilities before proposing new ones.

Claude Code must use existing components unless they are demonstrably unsuitable.

5.3 Backward Compatibility

Existing SolarPro users, projects, designs, BOQs, contracts, reports, assets and marketplace records must continue to work.

5.4 Enterprise Multi-Tenancy

Every enterprise organisation must have logically isolated data.

Use the repository’s current tenancy strategy. Where row-level security exists, extend it correctly. Where tenancy is enforced through service-layer filters, preserve and strengthen those controls.

5.5 Security by Default

All programme resources must be tenant-scoped, role-controlled, validated and auditable.

5.6 Configurable Workflows

Approvals, stages, roles, templates and programme types must not be embedded as inflexible code constants.

5.7 Scalable Portfolio Operations

The design must support thousands of beneficiaries and projects without loading all records into memory or rendering excessively large pages.

5.8 Idempotent Bulk Operations

Bulk beneficiary imports, project generation, BOQ aggregation and procurement consolidation must be safely repeatable.

5.9 Traceability

The system must retain traceability from a programme decision to programme template, beneficiary, generated project, design revision, BOQ, procurement package, contractor, installation, test, asset and operational record.

5.10 Human Approval

AI agents may recommend, calculate, draft, classify and monitor, but material programme, engineering, financial, procurement, contract and commissioning decisions must support human review and approval.
