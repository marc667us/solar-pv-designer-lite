What Phase 7 cutover does

The 15 global tables (product catalog, monitor state, anonymous webforms, admin settings, etc.) carry two policies today:

  • <table>_tenant_isolation — the parallel-run policy. Admits any row whose tenant_id is NULL OR matches the request's tenant GUC.
  • <table>_global_<intent> — the strict policy added by migration 015. Public-read tables admit any SELECT, anon-insert tables admit any INSERT, admin-only tables require the app.current_role = 'admin' GUC.

Phase 7 cutover drops the parallel-run policy on each of the 15 global tables. From that point on the strict policies alone decide who can see or modify a row. The app keeps working because the strict policies deliberately mirror the access patterns each table needs.

Cutover migration (not yet shipped)

Once every check above is PASS, the operator drafts migrations/017_phase7_drop_parallel_run.sql with a DROP POLICY <table>_tenant_isolation ON <table> statement per global table, wraps it in a workflow with the standard dry-run gate, runs the dry-run, then commits with confirm=PHASE7_DROP_PARALLEL.

What is NOT on this page: a button that runs the cutover. Policy drops cannot be undone without a separate restore migration; the cutover stays a manual, audited workflow trigger.
