I. Management's Assertion

Management of SolarPro Global is responsible for designing, implementing, and operating the controls described in this report to provide reasonable assurance that the service commitments to user entities and the applicable Trust Services Criteria (Security, Availability, Confidentiality) are met. The criteria selected are the AICPA Trust Services Criteria (2017).

Based on the controls described and tested in Sections III and IV, management asserts the controls were suitably designed as of the reporting date. Operating effectiveness over a 6+ month period will be established by an external Service Auditor in a future Type II engagement.

II. Description of the Service Organization's System

A. Services provided

SolarPro Global is a SaaS platform supporting engineering design of residential/commercial/industrial PV solar systems, BOM/BOQ generation with hierarchical project/building/floor structures, a multi-vendor Marketplace + RFQ workflow, multi-tenant project workspaces with role-based access (admin / owner / technical / sales / supplier / procurement specialist), and proposal generation.

B. Principal service commitments

  • Confidentiality of customer designs, BOMs, BOQs, and pricing intelligence within the tenant boundary.
  • Availability of the SaaS platform during reasonable business hours.
  • Integrity of engineering calculations, financial computations, and audit trails.
  • Compliance with applicable contractual data-protection obligations.

C. System components

Infrastructure
  • Compute: Render PaaS (prod)
  • Database: Render-managed PostgreSQL; SQLite (dev only)
  • IdP: Keycloak at auth.aiappinvent.com
  • CDN/DNS: Cloudflare
Software
  • Python 3.12 + Flask + Waitress / Gunicorn
  • Authlib OIDC client; PyJWT; JWKS cached
  • psycopg2; Postgres RLS on tenant tables
  • markdown-pdf for documents; Brevo for email
  • GitHub Actions security CI (semgrep, pip-audit, bandit, gitleaks)
People
  • Service Org Owner (founder; CEO equivalent)
  • Software Engineering function
  • Internal SOC 2 Compliance function
Procedures
  • Change mgmt via Git PRs + security CI
  • Production migrations gated by confirm-tokens
  • Access mgmt via Keycloak realm roles
  • Backups: nightly pg_dump 03:00 UTC
  • Evidence collection: nightly 03:30 UTC

D. Boundaries

In scope: SolarPro Flask app, Render-managed Postgres, Keycloak IdP, GitHub repo + workflows. Carved out (subservice orgs): Render PaaS, Brevo, Paystack, Stripe, Cloudflare, OpenRouter / Ollama / GitHub Models inference.

E. Identified risks & mitigations

RiskMitigation
Unauthorized cross-tenant data access Postgres RLS; tenant_id on tenant-owned tables; M1.7 tenant-context hooks
Authentication compromise Keycloak OIDC + PKCE S256; 14-day rollback workflow; MFA proposal drafted
Data integrity / orphan rows FOREIGN KEY constraints across owned-child + user-owned chains (migration 013)
Data loss Nightly pg_dump backup (M4.1); 30-day artefact retention
Vendor compromise semgrep + pip-audit + bandit + gitleaks on every push
Service outage Render auto-restart; /api/health/* for external uptime monitoring

F. Complementary user entity controls

  • Establish strong-password + MFA practices on the Keycloak account.
  • Recertify role assignments for the organization periodically.
  • Promptly notify SolarPro support of suspected unauthorized access.
  • Logically separate production and test workspaces.
